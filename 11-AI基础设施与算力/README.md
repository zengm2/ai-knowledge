# 11 - AI 基础设施与算力

> 如果说算法是 AI 的"大脑"，那么算力就是 AI 的"心脏"。没有强大的计算基础设施，再好的模型架构也只是纸上谈兵。本章系统介绍支撑 AI 发展的硬件、软件和云计算基础设施。

## 11.1 AI 计算的核心硬件

### 11.1.1 GPU：AI 训练的主力军

在深度学习出现之前，大多数计算任务都运行在 CPU 上。CPU 擅长处理复杂的逻辑控制和串行任务——就像一位全能型教授，什么都能做，但一次只能专注一件事。

GPU（Graphics Processing Unit，图形处理单元）最初是为游戏渲染设计的。游戏画面中每个像素的颜色计算都是相互独立的，因此 GPU 天生就是为**大规模并行计算**而生的。一块现代 GPU 拥有数千个计算核心，可以同时处理成千上万个简单的数学运算。

深度学习的核心操作——矩阵乘法——恰好具有高度并行的特点。一次前向传播涉及数十亿次浮点运算，而这些运算大多可以并行执行。这就是 GPU 在 AI 领域大放异彩的根本原因。

**打个比方：** CPU 像一辆跑车——速度快但只能载几个人；GPU 像一辆公交车——单人速度一般，但一次能载几十人。深度学习训练就像"早高峰上班"，需要同时运送大量乘客，公交车显然更合适。

| 对比维度 | CPU | GPU |
|---------|-----|-----|
| 核心数量 | 4-128 个高性能核心 | 数千至上万个简单核心 |
| 时钟频率 | 较高（3-5 GHz） | 较低（1-2 GHz） |
| 擅长任务 | 复杂逻辑、串行计算 | 大规模并行数值计算 |
| 内存带宽 | 较低（~100 GB/s） | 极高（~3 TB/s，如 H100） |
| AI 适用性 | 数据预处理、推理小模型 | 模型训练、大模型推理 |

NVIDIA 在 AI 计算领域占据绝对主导地位，其市场份额超过 90%。这不仅因为硬件性能出色，更因为 CUDA 软件生态的深度绑定——几乎所有主流深度学习框架都以 CUDA 为第一优先级进行优化。

## 11.2 CUDA 与 AI 软件栈

### 11.2.1 CUDA 生态系统

CUDA（Compute Unified Device Architecture）是 NVIDIA 于 2006 年推出的并行计算平台和编程模型。它允许开发者使用类 C 语言直接编写 GPU 程序，是 NVIDIA 建立 AI 算力护城河的关键所在。

CUDA 生态的核心组件包括：

| 组件 | 功能 | 重要性 |
|------|------|--------|
| CUDA Toolkit | GPU 编程基础工具包（编译器、运行时） | 一切的基础 |
| cuDNN | 深度神经网络加速库（卷积、RNN 等） | PyTorch/TensorFlow 的底层依赖 |
| NCCL | 多 GPU 通信库（AllReduce 等集合操作） | 分布式训练必备 |
| TensorRT | 推理优化引擎（算子融合、量化） | 生产环境推理加速 |
| cuBLAS | GPU 线性代数库（矩阵运算加速） | 所有矩阵运算的基础 |
| Triton（OpenAI） | Python 风格的 GPU 编程语言 | 降低 GPU 编程门槛 |

NVIDIA 的护城河并不仅仅是硬件性能——而是十几年积累的 CUDA 软件生态。几乎所有深度学习框架（PyTorch、TensorFlow、JAX）都优先支持 CUDA，数以百万计的开发者已经形成了对 CUDA 的依赖。即使竞争对手的硬件性能接近，迁移成本也极高。

### 11.2.2 PyTorch 的硬件抽象

PyTorch 通过简洁的 API 将底层硬件差异封装起来，开发者只需几行代码就能在 CPU 和 GPU 之间切换：

```python
import torch

# 检查 GPU 是否可用
print(f"CUDA 可用: {torch.cuda.is_available()}")
print(f"GPU 数量: {torch.cuda.device_count()}")
if torch.cuda.is_available():
    print(f"GPU 型号: {torch.cuda.get_device_name(0)}")

# 创建张量并移动到 GPU
x = torch.randn(1000, 1000)           # 默认在 CPU 上
x_gpu = x.to("cuda")                   # 移动到 GPU
# 或者直接在 GPU 上创建
y_gpu = torch.randn(1000, 1000, device="cuda")

# GPU 上的矩阵乘法（自动调用 cuBLAS）
result = x_gpu @ y_gpu

# 模型移动到 GPU
model = MyModel()
model = model.to("cuda")               # 整个模型移至 GPU

# 混合精度训练（自动选择 FP16/FP32）
from torch.cuda.amp import autocast, GradScaler
scaler = GradScaler()

with autocast():
    output = model(input_data)
    loss = criterion(output, target)

scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

这种设计的好处是：模型代码与硬件解耦，同一份训练脚本可以在笔记本 CPU、单块 GPU 或多卡集群上运行，只需修改设备参数。

### 11.2.3 替代方案

虽然 CUDA 是当前的事实标准，但竞争者正在逐步追赶：

- **ROCm（AMD）**：AMD 的开源 GPU 计算平台。PyTorch 已原生支持 ROCm，MI300X 在部分场景下性能可与 H100 竞争。主要挑战在于软件成熟度和第三方库支持。
- **Metal / MLX（Apple）**：Apple 的 Metal API 用于 GPU 计算，MLX 是 Apple Silicon 原生的机器学习框架，语法类似 NumPy/PyTorch。适合 Mac 用户进行本地推理和小规模实验。
- **OpenCL**：跨平台的开放标准，理论上支持各种硬件，但在深度学习领域的优化远不如 CUDA。
- **SYCL / oneAPI（Intel）**：Intel 推出的跨架构编程模型，旨在统一 CPU、GPU、FPGA 的编程接口。生态尚处早期。

## 11.3 分布式训练

### 11.3.1 为什么需要分布式训练

以 GPT-3 为例，其参数量为 1750 亿。如果以 FP16（每个参数 2 字节）存储模型参数，仅参数本身就需要约 350 GB 显存——而一块 H100 只有 80 GB。加上优化器状态（Adam 需要额外存储一阶和二阶矩，约为参数量的 2-3 倍）、梯度和激活值，单卡训练大模型是完全不可能的。

除了显存限制，**训练时间**也是关键因素。Llama 3 70B 的训练在 16,384 块 H100 上进行了约 54 天。如果只用 1 块 GPU，理论上需要超过 240 万天（约 6600 年）。分布式训练是唯一的出路。

### 11.3.2 数据并行

数据并行（Data Parallelism）是最直观的分布式策略：

1. 每块 GPU 持有模型的**完整副本**
2. 训练数据被均匀分割到各 GPU
3. 每块 GPU 独立计算前向传播和梯度
4. 通过 AllReduce 操作同步梯度，确保所有 GPU 上的模型参数保持一致

```
GPU 0: 数据批次 A → 计算梯度 ─┐
GPU 1: 数据批次 B → 计算梯度 ─┼→ AllReduce 同步 → 更新参数
GPU 2: 数据批次 C → 计算梯度 ─┤
GPU 3: 数据批次 D → 计算梯度 ─┘
```

PyTorch 提供了两种主要的数据并行实现：

- **DDP（DistributedDataParallel）**：标准的数据并行方案，每卡持有完整的模型和优化器状态。简单高效，但显存占用大。
- **FSDP（Fully Sharded Data Parallel）**：将模型参数、梯度和优化器状态分片到各 GPU，需要时再聚合。大幅降低单卡显存需求，代价是增加通信开销。

### 11.3.3 模型并行

当模型大到单卡放不下时，就需要将模型本身拆分到多块 GPU 上：

**张量并行（Tensor Parallelism）**

将单个层的计算（如一个大矩阵乘法）拆分到多块 GPU 上。例如，一个 `[4096, 16384]` 的权重矩阵可以沿列切分为 4 块 `[4096, 4096]`，分别放在 4 块 GPU 上计算，最后合并结果。

- 优点：每一层的计算都能并行加速
- 缺点：需要高速互联（如 NVLink），通信延迟敏感

**流水线并行（Pipeline Parallelism）**

将模型的不同层分配到不同 GPU 上，数据像工厂流水线一样依次流过各 GPU：

```
GPU 0: 第 1-20 层  →  GPU 1: 第 21-40 层  →  GPU 2: 第 41-60 层  →  GPU 3: 第 61-80 层
```

- 优点：通信量较小，对互联带宽要求低于张量并行
- 缺点：存在"流水线气泡"——前面的 GPU 在等后面 GPU 完成计算时处于空闲状态

### 11.3.4 混合并行策略

实际的大模型训练通常组合使用多种并行策略，这被称为 **3D 并行**：

```
                    ┌──────────────────────────────┐
                    │        数据并行（DP）          │
                    │  将数据分到多个模型副本组       │
                    └──────────────────────────────┘
                                  │
                    ┌─────────────┼─────────────┐
                    ▼                           ▼
            ┌──────────────┐            ┌──────────────┐
            │  流水线并行    │            │  流水线并行    │
            │ 模型按层切分   │            │ 模型按层切分   │
            └──────────────┘            └──────────────┘
                    │                           │
               ┌────┼────┐                 ┌────┼────┐
               ▼    ▼    ▼                 ▼    ▼    ▼
              TP   TP   TP               TP   TP   TP
          （张量并行：层内切分）        （张量并行：层内切分）
```

**DeepSpeed ZeRO**（Zero Redundancy Optimizer）是微软开发的显存优化技术，分为三个阶段：

| 阶段 | 分片内容 | 显存节省 | 通信开销 |
|------|---------|---------|---------|
| ZeRO-1 | 仅优化器状态 | ~4x | 低 |
| ZeRO-2 | 优化器状态 + 梯度 | ~8x | 中等 |
| ZeRO-3 | 优化器状态 + 梯度 + 参数 | ~N 倍（N=GPU 数） | 较高 |

### 11.3.5 主要框架

| 框架 | 开发者 | 核心特点 | 适用场景 |
|------|--------|---------|---------|

## 11.4 模型推理优化

训练完成后，模型需要在生产环境中高效运行。推理优化的目标是：在保持模型质量的前提下，尽可能降低延迟、提高吞吐量、减少成本。

### 11.4.1 量化技术

量化是将模型参数从高精度浮点数转换为低精度表示的技术，从而减少显存占用和加速计算：

| 精度格式 | 每参数字节 | 70B 模型大小 | 质量影响 | 典型用途 |
|---------|-----------|-------------|---------|---------|
| FP32 | 4 字节 | ~280 GB | 基准 | 科研、精确计算 |
| FP16/BF16 | 2 字节 | ~140 GB | 几乎无损 | 标准训练和推理 |
| INT8 | 1 字节 | ~70 GB | 轻微下降 | 生产推理 |
| INT4 | 0.5 字节 | ~35 GB | 可接受的下降 | 消费级 GPU 部署 |

主流量化方法：

- **GPTQ**：基于逐层量化的 post-training 方法，需要校准数据集。量化后的模型可直接使用 GPU 高效推理。
- **AWQ（Activation-aware Weight Quantization）**：根据激活值的分布来选择性地保护重要权重，通常比 GPTQ 保持更好的质量。
- **GGUF（llama.cpp 格式）**：专为 CPU 和混合 CPU/GPU 推理设计的量化格式，支持多种量化级别（Q4_K_M, Q5_K_M, Q8_0 等），在消费级硬件上非常流行。

### 11.4.2 推理加速技术

现代 LLM 推理有多种关键优化技术：

- **KV Cache**：自回归生成中，已处理 token 的 Key 和 Value 可以缓存起来，避免重复计算。这是最基本的推理优化，几乎所有框架都会默认启用。
- **Flash Attention**：由斯坦福 Tri Dao 提出，通过重新组织注意力计算的内存访问模式，将标准注意力的显存占用从 O(N^2) 降低到 O(N)，同时速度提升 2-4 倍。
- **PagedAttention**：vLLM 引入的技术，将 KV Cache 像操作系统管理内存页一样进行分页管理，解决了显存碎片化问题，使批处理吞吐量提升 2-4 倍。
- **投机解码（Speculative Decoding）**：用一个小模型快速"猜测"多个后续 token，再用大模型并行验证。被接受的 token 无需重新生成，从而加速推理过程。
- **前缀缓存（Prefix Caching）**：对于共享相同系统提示的请求，缓存公共前缀的 KV Cache，避免重复计算。这在多轮对话和 API 服务中非常有效。

### 11.4.3 推理框架

| 框架 | 开发者 | 核心优势 | 硬件支持 | 适用场景 |
|------|--------|---------|---------|---------|
| vLLM | UC Berkeley | PagedAttention、高吞吐 | NVIDIA/AMD GPU | 服务端大规模推理 |
| TensorRT-LLM | NVIDIA | 极致 NVIDIA 优化 | 仅 NVIDIA GPU | NVIDIA 环境最佳性能 |
| llama.cpp | 社区 | CPU 推理、GGUF 量化 | CPU/GPU/Metal | 本地部署、边缘设备 |
| Ollama | 社区 | 一键部署、用户友好 | CPU/GPU/Metal | 个人本地使用 |
| SGLang | UC Berkeley | 结构化生成优化 | NVIDIA GPU | 需要 JSON 输出等场景 |
| TGI | Hugging Face | HF 生态集成 | NVIDIA/AMD GPU | Hugging Face 用户 |

### 11.4.4 边缘部署

随着量化技术的进步，在手机和笔记本电脑上运行 AI 模型已成为现实：

- **Apple 设备**：M 系列芯片的统一内存架构天然适合 LLM 推理。使用 MLX 框架或 llama.cpp 的 Metal 后端，M2/M3/M4 MacBook 可以流畅运行 7B-13B 的量化模型。
- **Android 设备**：通过 Android NNAPI 或 MediaPipe，可以在高端手机上运行小型模型（如 Gemma 2B）。Google 的 Gemini Nano 就是专为端侧推理设计的。
- **Windows PC**：通过 DirectML 或 ONNX Runtime，可以利用各种 GPU（NVIDIA、AMD、Intel）进行本地推理。

边缘部署的关键考量：模型通常需要量化到 INT4 才能在消费级设备上获得可接受的速度，这会带来一定的质量损失。选择模型时需要在大小、速度和质量之间做权衡。

| Vast.ai | 各类 GPU | 市场定价，极低 | P2P 市场、价格最低 |

### 11.5.2 GPU 租赁策略

云 GPU 的计费模式多种多样，选择合适的策略可以大幅降低成本：

- **按需实例（On-Demand）**：随用随付，最灵活但最贵。适合短期实验和不确定的工作负载。
- **预留实例（Reserved）**：承诺使用 1-3 年，价格可降低 40-60%。适合长期稳定的训练任务。
- **竞价实例（Spot/Preemptible）**：使用闲置算力，价格可低至按需的 30-50%，但可能随时被中断。适合可断点续训的任务。

**成本优化实战建议：**

1. 开发调试阶段使用小型 GPU 或 CPU，确认代码无误后再切换到大卡
2. 训练时务必实现 checkpoint 保存机制，以支持竞价实例的断点续训
3. 推理服务根据流量波动使用自动扩缩容（autoscaling）
4. 对比多家平台的定价——同一型号 GPU 的价格差异可达 5-10 倍

### 11.5.3 Serverless AI 推理

如果只需要调用模型进行推理，而不想管理 GPU 服务器，Serverless 平台是一个好选择：

- **Replicate**：上传模型后获得 API 端点，按预测次数计费。支持自定义模型和热门开源模型。
- **Together AI**：提供主流开源模型的推理 API，兼容 OpenAI 格式，价格通常比 OpenAI 低很多。
- **Hugging Face Inference Endpoints**：一键部署 HF Hub 上的模型，支持自动扩缩容和自定义硬件配置。
- **Modal**：面向开发者的 Serverless GPU 平台，支持用 Python 代码定义计算任务，按秒计费。

Serverless 模式的核心优势是"零运维"——不需要配环境、管服务器、处理扩缩容，只需关注业务逻辑。但对于高吞吐量的持续服务，自建推理服务通常更经济。

## 11.6 AI 计算的成本经济学

### 11.6.1 训练成本

大模型的训练成本已成为 AI 领域的热门话题。以下是一些估算数据（基于公开信息和行业估算）：

| 模型 | 参数量 | 训练硬件 | 估算成本 | 训练时长 |
|------|--------|---------|---------|---------|
| GPT-3 (2020) | 175B | ~1,000 A100 | ~$500 万 | 约 34 天 |
| GPT-4 (2023) | 未公开（传闻 MoE） | ~25,000 A100 | ~$1 亿+ | 约 90 天 |
| Llama 3 70B (2024) | 70B | 16,384 H100 | ~$2,000 万 | 约 54 天 |
| DeepSeek-V3 (2024) | 671B MoE | 2,048 H800 | ~$550 万 | 约 60 天 |

DeepSeek-V3 的训练成本引发了行业震动——以不到 GPT-4 训练成本的 1/10，训练出了性能可比的模型。这得益于 MoE 架构的高效性、FP8 混合精度训练，以及工程优化的持续进步。

**趋势：** 虽然顶级模型的绝对训练成本仍在增长，但"单位性能的训练成本"在快速下降。每年同等水平模型的训练成本约下降 2-3 倍。

### 11.6.2 推理成本

对于大多数 AI 应用来说，推理成本远大于训练成本（训练只做一次，推理每天都在进行）。2024 年，主要 API 提供商之间爆发了激烈的价格战：

| 提供商 | 模型 | 输入价格（/百万 token） | 输出价格（/百万 token） |
|--------|------|----------------------|----------------------|
| OpenAI | GPT-4o | $2.50 | $10.00 |
| OpenAI | GPT-4o-mini | $0.15 | $0.60 |
| Anthropic | Claude 3.5 Sonnet | $3.00 | $15.00 |
| Anthropic | Claude 3.5 Haiku | $0.80 | $4.00 |
| DeepSeek | DeepSeek-V3 | $0.27 | $1.10 |
| DeepSeek | DeepSeek-R1 | $0.55 | $2.19 |
| Google | Gemini 1.5 Flash | $0.075 | $0.30 |

> 💡 **注意：** 以上价格为截至 2024 年末的参考数据，API 价格变化频繁。DeepSeek 和 Google 的激进定价正在推动整个行业的成本下降。

### 11.6.3 个人开发者的算力方案

针对不同预算和需求，个人开发者可以参考以下算力方案：

**场景一：学习和实验（零预算）**
- Google Colab 免费版：T4 GPU，15GB 显存，足以运行 7B 量化模型
- Kaggle Notebooks：每周 30 小时 GPU 时间（T4/P100）
- 适合入门学习、小规模实验

**场景二：本地推理和微调（一次性投资）**
- RTX 4090（24GB）：约 ¥13,000，能运行大多数开源模型的量化版本
- 对于 Mac 用户，M2/M3/M4 Pro/Max 的统一内存（32-128GB）也是不错的选择
- 适合日常推理、LoRA 微调、小规模实验

**场景三：中等规模训练（按需付费）**
- 云 GPU 按小时租用（AutoDL、Lambda Cloud 等）
- 单卡 A100 微调通常几小时到几十小时即可完成
- 月均费用：几百到几千元

**场景四：大规模预训练（团队/企业）**
- 多卡集群或云端大规模实例
- 预留实例 + 竞价实例组合使用
- 月均费用：数万到数百万元

## 11.7 学习建议与延伸阅读

掌握 AI 基础设施知识不仅能帮助你更好地理解模型训练和部署的全流程，还能在实际项目中做出更明智的技术决策。以下是推荐的学习路径：

**入门级：**
- 学会使用 PyTorch 的 GPU 计算（`.to("cuda")`、混合精度训练）
- 了解量化概念，尝试用 Ollama 或 llama.cpp 在本地运行开源模型
- 使用 Google Colab 完成一次简单的模型微调

**进阶级：**
- 学习 CUDA 编程基础：推荐 NVIDIA 官方的 [CUDA Programming Guide](https://docs.nvidia.com/cuda/cuda-c-programming-guide/)
- 掌握分布式训练：从 PyTorch DDP 入手，逐步了解 FSDP 和 DeepSpeed
- 实践推理优化：部署一个 vLLM 服务，对比不同量化方法的效果

**深入级：**
- 阅读 Flash Attention 论文（Tri Dao, 2022）理解 GPU 内存层级优化
- 研究 Megatron-LM 的 3D 并行实现
- 了解 GPU 架构细节：SM（Streaming Multiprocessor）、Warp 调度、显存层级

**推荐资源：**
- 《Programming Massively Parallel Processors》—— GPU 编程经典教材
- NVIDIA 官方博客（developer.nvidia.com/blog）—— 最新技术文章
- Hugging Face 文档中的分布式训练章节
- DeepSpeed 官方文档和教程
- llama.cpp 项目 Wiki —— 量化和本地部署实践
