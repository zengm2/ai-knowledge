# Q: 检索增强生成（RAG）是个什么东西？

> **关键词**：RAG、检索增强生成、Retrieval-Augmented Generation、向量检索、Embedding、知识库问答、幻觉

**简短回答**：RAG = 让大模型先"查资料"再"回答问题"。它在推理时给模型提供外部知识作为参考，解决了大模型知识过时、容易幻觉、缺乏私有知识的核心痛点。

---

## 详细解答

### 为什么需要 RAG？

| 问题 | 说明 | RAG 怎么解决 |
|------|------|-------------|
| **知识过时** | 模型训练数据有截止日期 | 从最新的文档/数据库中检索 |
| **幻觉** | 模型会编造事实 | 基于检索到的真实内容回答 |
| **缺乏私有知识** | 模型不知道公司内部文档 | 检索自己的知识库 |
| **上下文长度有限** | 不能把所有资料塞进 prompt | 只检索最相关的片段 |

### 工作流程

```
用户提问: "公司的年假政策是什么？"
        ↓
① 检索（Retrieval）
   把问题转成向量 → 在向量数据库中搜索最相似的文档 → 返回 Top-K 相关片段
        ↓
② 增强（Augmented）
   把检索到的内容拼接到 prompt 中作为参考资料
        ↓
③ 生成（Generation）
   大模型基于检索到的真实内容生成回答
        ↓
"根据公司规定，入职满1年可享有5天年假..."
```

### 核心组件

#### 1. 文档处理（离线准备阶段）

```
原始文档（PDF、Word、网页等）
    ↓ 文档解析（提取文本）
    ↓ 分块（Chunking）—— 把长文档切成小段
    ↓ 向量化（Embedding）—— 用模型把文本转成向量
    ↓ 存入向量数据库（Pinecone、Milvus、Chroma、FAISS 等）
```

#### 2. 向量检索（核心机制）

```
关键词搜索：查询 "年假" → 只能匹配含 "年假" 的文档
向量搜索：  查询 "年假" → 也能匹配 "带薪休假"、"annual leave" 等语义相近内容
```

原理：文本转成高维向量，语义相近的文本向量距离也近，用余弦相似度找最近的向量。

#### 3. Prompt 组装

```python
prompt = f"""根据以下参考资料回答用户的问题。
如果参考资料中没有相关信息，请说明你不确定。

参考资料：
{retrieved_documents}

用户问题：{user_question}
"""
```

### 和直接用大模型的区别

| 对比 | 纯大模型 | RAG |
|------|---------|-----|
| **知识来源** | 仅训练时学到的 | 训练知识 + 实时检索的外部知识 |
| **时效性** | 有截止日期 | 知识库更新即可获取最新信息 |
| **准确性** | 容易幻觉 | 有据可查，可追溯来源 |
| **私有数据** | 不了解 | 可以接入企业私有知识库 |
| **可解释性** | 黑盒 | 可以展示引用来源 |

### 实际应用场景

| 场景 | 说明 |
|------|------|
| **企业知识库问答** | 查内部制度、产品文档、操作手册 |
| **客服系统** | 基于产品文档自动回答用户问题 |
| **法律/医疗咨询** | 检索法条、病例、药品说明书 |
| **代码助手** | 检索项目代码库和文档辅助开发 |
| **AI 搜索引擎** | 如 Perplexity，先搜索再生成 |

### 挑战和进阶方向

| 挑战 | 说明 |
|------|------|
| **分块策略** | 切太大不精确，切太小丢上下文 |
| **检索质量** | 检索不到正确内容，生成再好也没用 |
| **多跳推理** | 答案分散在多个文档中 |

进阶方向：Advanced RAG（查询改写、Reranking）、Graph RAG（结合知识图谱）、Agentic RAG（Agent 自主决定检索策略）。

### 一句话总结

RAG 不改变模型本身，而是在推理时给模型提供外部知识作为参考，是当前 AI 落地应用中最重要的技术模式之一。
