# Q: 提示工程是需要用户来做还是需要专员？和提示词有什么区别？

> **关键词**：提示工程、Prompt Engineering、提示词、Prompt、System Prompt、Few-shot、Chain-of-Thought

**简短回答**：提示词是你写给 AI 的那段话，提示工程是研究怎么写好这段话的方法论。普通用户掌握基础技巧就能提升效果；专业场景下需要专人负责提示工程的设计和优化。

---

## 详细解答

### 提示工程 vs 提示词

| 对比 | 提示词（Prompt） | 提示工程（Prompt Engineering） |
|------|-----------------|-------------------------------|
| **本质** | 一段具体的文本输入 | 一套方法论和技术体系 |
| **类比** | 一道菜 | 烹饪技术 |
| **举例** | "帮我写一封邮件" | 研究怎么写这句话能让输出质量最高 |
| **门槛** | 人人都在做（用 AI 就在写 prompt） | 需要系统学习和实践 |

```
提示词 = 你写的那句话
提示工程 = 研究怎么把话说得更好的学问
```

### 谁来做提示工程？

#### 1. 普通用户 —— 日常使用级别

每个使用 AI 工具的人，本质上都在做最基础的提示工程：

```
❌ 差的提示词：  "写个邮件"
✅ 好的提示词：  "帮我写一封给客户的道歉邮件，语气诚恳专业，
                 因为发货延迟了3天，150字以内"
```

普通用户学会基本技巧就能显著提升效果：说清角色、任务、格式、约束，给例子，分步骤提问。

#### 2. 专业角色 —— 提示工程师（Prompt Engineer）

在企业和产品开发中，提示工程是专门的工作甚至岗位：

| 工作内容 | 说明 |
|---------|------|
| **System Prompt 设计** | 设计产品背后用户看不到的系统提示词 |
| **提示词模板化** | 设计可复用的 prompt 模板 |
| **效果评估** | 系统测试不同 prompt 的输出质量，做 A/B 测试 |
| **防护设计** | 防止 prompt 注入攻击 |
| **与业务结合** | 把领域知识嵌入 prompt |
| **RAG/Agent 中的 prompt** | 在复杂 AI 系统中设计各环节的提示词 |

### 实际分工

```
普通用户：           专业提示工程师/开发者：

"帮我总结这篇文章"     设计系统 prompt：
                      "你是一个专业的内容分析师，
                       请按以下结构输出：
                       1. 核心论点（1句话）
                       2. 关键论据（3-5条）
                       3. 作者立场
                       4. 潜在偏见
                       输出格式：Markdown ..."
                      ↑ 这段用户看不到，是产品内置的
```

### 提示工程的核心技术

| 技术 | 说明 | 举例 |
|------|------|------|
| **Zero-shot** | 直接提问，不给例子 | "翻译成英文：今天天气很好" |
| **Few-shot** | 给几个示例再提问 | "猫→cat，狗→dog，鱼→？" |
| **Chain-of-Thought** | 引导模型分步推理 | "请一步步思考这个数学题" |
| **角色设定** | 指定模型身份 | "你是一个资深 Python 工程师" |
| **格式约束** | 要求特定输出格式 | "以 JSON 格式返回" |
| **Self-consistency** | 多次采样取最一致的答案 | 同一问题问 5 次取多数 |
| **ReAct** | 交替"思考"和"行动" | Agent 场景的核心 prompt 模式 |

### 一句话总结

人人都在做 prompt，但做得好不好差距很大。普通用户掌握基础技巧即可大幅提升效果，专业场景需要专人系统化设计和优化。
