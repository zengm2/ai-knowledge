# 06 - 大语言模型 (LLM)

> 大语言模型（Large Language Model, LLM）是当前人工智能领域最具变革性的技术方向。从 ChatGPT 横空出世到多模态大模型遍地开花，LLM 正在深刻改变我们与计算机交互的方式。本章将系统介绍 LLM 的核心概念、发展历程、训练方法、应用范式以及部署优化策略。

---

## 6.1 什么是大语言模型

### 6.1.1 定义

大语言模型（LLM）是指**基于 Transformer 架构、在海量文本数据上进行预训练的大规模神经网络语言模型**。它通过学习文本中的统计规律和语义关系，获得了强大的语言理解与生成能力。与传统的 NLP 模型不同，LLM 展现出了令人惊叹的**涌现能力（Emergent Abilities）**——当模型规模达到一定阈值时，会突然获得推理、编程、翻译等之前未被显式训练过的能力。

从本质上说，LLM 是一个**概率模型**：给定一段文本前缀，它会预测下一个最有可能出现的 token（词元）。但正是这个看似简单的"下一个词预测"任务，在足够大的规模下，催生出了通用人工智能的雏形。

### 6.1.2 "大"体现在哪里

LLM 的"大"主要体现在三个维度：

| 维度 | 说明 | 典型数量级 |
|------|------|-----------|
| **参数量（Parameters）** | 模型中可学习权重的数量 | 7B ~ 1.8T（70亿到1.8万亿） |
| **训练数据量（Data）** | 预训练使用的文本语料规模 | 数万亿 tokens（如 LLaMA 3 使用了 15T tokens） |
| **算力（Compute）** | 训练消耗的计算资源 | 数千到数万张 GPU，训练数周到数月 |

例如，GPT-3 拥有 1750 亿参数，GPT-4 据估计超过 1 万亿参数（MoE 架构），而 DeepSeek-V3 拥有 6710 亿参数。训练一个大型模型可能需要数千万美元的算力投入。

### 6.1.3 Scaling Law（缩放定律）

OpenAI 在 2020 年提出的 **Scaling Law** 是 LLM 发展的核心指导原则。其核心发现是：

- 模型性能（以交叉熵损失衡量）与**参数量 N**、**数据量 D**、**计算量 C** 之间存在幂律关系
- 三者的增长需要**协调匹配**，单独增加某一个维度的收益会递减
- Chinchilla 定律进一步指出：**最优策略是数据量和参数量等比例增长**

Scaling Law 给业界提供了明确的指引——只要持续投入更多资源，模型就能持续变得更强。这也解释了为什么各大科技公司都在疯狂扩大训练规模。

### 6.1.4 推理时计算与测试时缩放（Test-time Scaling）

2024 年下半年，一种新的范式正在兴起：**与其在训练时投入更多资源，不如在推理时让模型"多想一会儿"**。

- **o1/o3 模型**验证了这一范式：通过在推理时生成内部思维链（Chain-of-Thought），让模型进行多步推理，在数学、编程等复杂任务上获得了突破性提升
- **核心思想**：给模型更多的"思考时间"（推理时计算预算），让它能够探索多条推理路径、验证中间结果
- **意义**：这标志着 LLM 发展可能从"训练时缩放"向"推理时缩放"转变，为提升模型能力开辟了新的方向
- **局限性**：推理模型的延迟更高、成本更大，不适合所有应用场景（如实时对话）

这一范式转变表明，**模型的智能不仅来自训练阶段积累的知识，也来自推理阶段的动态计算**。

## 6.2 LLM 发展历程

### 6.2.1 GPT 系列：开创自回归范式

| 模型 | 年份 | 参数量 | 里程碑意义 |
|------|------|--------|-----------|
| **GPT-1** | 2018 | 1.17亿 | 首次证明了"预训练 + 微调"范式在 NLP 中的有效性 |
| **GPT-2** | 2019 | 15亿 | 展示了零样本学习能力，OpenAI 因担心滥用曾推迟发布 |
| **GPT-3** | 2020 | 1750亿 | 少样本学习能力震惊世界，催生了 Prompt Engineering |
| **ChatGPT** | 2022.11 | — | 基于 GPT-3.5，通过 RLHF 对齐，引爆全球 AI 热潮 |
| **GPT-4** | 2023.3 | ~1.8T（推测） | 多模态能力，推理能力大幅提升，可能采用 MoE 架构 |
| **GPT-4o** | 2024.5 | — | 原生多模态（文本、图像、音频统一处理），响应速度极快 |
| **GPT-4o mini** | 2024.7 | — | 轻量级多模态模型，成本极低，适合大规模应用 |
| **o1** | 2024.9 | — | 引入"推理时计算"范式，通过内部思维链进行深度推理 |
| **o3 / o4-mini** | 2025 | — | 推理能力进一步突破，在数学和编程竞赛中达到人类专家水平 |

### 6.2.2 BERT 系列：双向编码器

- **BERT**（2018）：Google 提出，基于 Transformer 编码器，通过掩码语言模型（MLM）进行预训练，在 NLU 任务上取得突破
- **RoBERTa**（2019）：Meta 改进了 BERT 的训练策略（更多数据、更长训练），性能显著提升
- **DeBERTa**（2020）：微软引入解耦注意力机制，在多个基准上超越 BERT 和 RoBERTa

BERT 系列主要用于**自然语言理解**任务（分类、实体识别、问答），而 GPT 系列主要用于**生成任务**。随着 LLM 的发展，生成式模型（GPT 范式）逐渐成为主流。

### 6.2.3 其他重要模型

- **T5**（2019）：Google 提出的"文本到文本"统一框架，将所有 NLP 任务统一为 seq2seq 格式
- **PaLM / PaLM 2**（2022-2023）：Google 的大规模模型，展示了强大的推理能力
- **Gemini**（2023-2024）：Google 最新的多模态大模型系列，与 GPT-4 正面竞争
- **Gemini 2.0**（2024.12）：原生多模态，支持视频理解和更强的推理能力
- **Claude**（2023-2025）：Anthropic 推出，以安全性和超长上下文（200K tokens）著称，Claude 3.5 Sonnet 以出色的代码能力闻名

### 6.2.4 开源模型的崛起

开源模型的快速发展是 LLM 领域最重要的趋势之一：

| 模型 | 开发者 | 特点 |
|------|--------|------|
| **LLaMA / LLaMA 2 / LLaMA 3** | Meta | 开源生态的基石，Llama 3.1 405B 是最大的开源模型，推动了整个社区的繁荣 |
| **Mistral / Mixtral** | Mistral AI | 小参数量大能力，MoE 架构的典范 |
| **Qwen（通义千问）** | 阿里巴巴 | 中文能力强劲，Qwen 2.5 系列覆盖 0.5B-72B，代码和数学能力突出 |
| **DeepSeek** | 深度求索 | DeepSeek-V3（6710亿参数）和 R1 在推理能力上表现突出，训练成本极低，引发行业震动 |
| **ChatGLM** | 智谱 AI | 中英双语对话模型，国内最早的开源 LLM 之一 |
| **Yi（零一万物）** | 零一万物 | 李开复团队，中英文能力均衡 |
| **Phi-4** | 微软 | 小参数量（14B）但能力突出，适合边缘部署和资源受限场景 |

**开源 vs 闭源**的格局正在不断演变。闭源模型（GPT-4、Claude、Gemini）在能力上仍有优势，但开源模型的差距在迅速缩小。特别是 DeepSeek-R1 的出现，证明了开源模型在推理任务上可以达到甚至超越闭源模型的水平。

## 6.3 LLM 的训练过程

现代 LLM 的训练通常遵循**三阶段流程**：预训练 → 有监督微调 → 人类对齐。

### 6.3.1 预训练（Pre-training）

**目标**：在海量无标注文本上学习语言的通用规律。

- **训练任务**：自回归语言建模（Autoregressive LM）——给定前面的 tokens，预测下一个 token
- **数据来源**：互联网网页（Common Crawl）、书籍、学术论文、代码仓库、百科全书等
- **数据规模**：通常达到数万亿 tokens
- **训练成本**：极其昂贵，如 LLaMA 3 405B 使用 16,384 张 H100 GPU 训练
- **关键技术**：分布式训练（数据并行、张量并行、流水线并行）、混合精度训练（BF16/FP16）、Flash Attention

预训练后的模型被称为**基座模型（Base Model）**，它具备了强大的语言能力，但还不擅长遵循指令或进行对话。

### 6.3.2 有监督微调（Supervised Fine-Tuning, SFT）

**目标**：让模型学会遵循指令、进行对话。

- **数据格式**：（指令, 输出）配对数据，如用户问题和理想回答
- **数据规模**：通常几万到几十万条高质量样本
- **关键点**：数据质量远比数量重要！少量精心标注的数据效果可能优于大量低质量数据
- **典型数据集**：FLAN、Alpaca、ShareGPT、OpenAssistant

SFT 之后，模型就变成了我们熟悉的"聊天助手"形态。

### 6.3.3 RLHF（Reinforcement Learning from Human Feedback）

**目标**：进一步对齐人类偏好，让模型输出更有用、更安全、更诚实。

RLHF 分为两个阶段：

1. **训练奖励模型（Reward Model, RM）**：
   - 让模型对同一问题生成多个回答
   - 人类标注员对这些回答进行偏好排序
   - 训练一个奖励模型来预测人类的偏好分数

2. **用强化学习优化策略模型**：
   - 使用 PPO（近端策略优化）算法
   - 让语言模型（策略模型）生成回答
   - 奖励模型打分
   - 根据分数更新语言模型的参数
   - 同时用 KL 散度约束，防止模型偏离基座模型太远

RLHF 是 ChatGPT 成功的关键技术之一，但它的训练过程复杂、不稳定，且成本较高。

### 6.3.4 DPO（Direct Preference Optimization）

**目标**：简化 RLHF 流程，无需训练单独的奖励模型。

DPO 的核心思想是：将偏好学习问题转化为一个简单的**二元分类问题**。

- 给定一对（好回答, 差回答），直接优化模型使其更倾向于好回答
- 数学上等价于 RLHF，但实现简单得多
- 训练稳定性更好，超参数更少

DPO 因其简洁性已被广泛采用。其变体包括 IPO、KTO、ORPO 等，各自针对不同场景进行了优化。2024 年的实践表明，**SFT + DPO** 的组合已成为训练高质量对话模型的标准流程。

此外值得关注的对齐方法还有：
- **GRPO（Group Relative Policy Optimization）**：DeepSeek 提出的方法，在群组层面优化策略，用于训练 DeepSeek-R1
- **Constitutional AI**：Anthropic 提出，使用一组"宪法原则"指导模型行为，减少对大量人类反馈的依赖

## 6.4 提示工程（Prompt Engineering）

### 6.4.1 什么是 Prompt

**Prompt（提示）** 是用户输入给 LLM 的文本指令。LLM 的输出质量在很大程度上取决于 Prompt 的质量。提示工程（Prompt Engineering）就是**设计和优化 Prompt 以获得更好输出**的技术。

一个完整的 Prompt 通常包含：
- **角色设定**：告诉模型它是谁（"你是一位资深Python工程师"）
- **任务描述**：要求模型做什么
- **上下文信息**：提供相关的背景知识
- **输出格式**：指定输出的格式要求（JSON、Markdown、表格等）
- **约束条件**：限制模型的行为边界

### 6.4.2 Zero-Shot vs Few-Shot Prompting

- **Zero-Shot（零样本）**：直接给出任务描述，不提供示例
  ```
  请将以下英文翻译为中文：Hello, how are you?
  ```

- **Few-Shot（少样本）**：提供几个输入-输出示例，帮助模型理解任务模式
  ```
  请将英文翻译为中文：
  English: Good morning → 中文: 早上好
  English: Thank you → 中文: 谢谢
  English: Hello, how are you? → 中文:
  ```

Few-Shot 在复杂任务或特定格式要求时特别有效。

### 6.4.3 Chain-of-Thought（CoT）思维链

CoT 是一种让模型**逐步推理**的提示技术，对数学、逻辑推理任务效果显著：

- **手动 CoT**：在示例中展示推理过程
- **零样本 CoT**：只需添加"让我们一步一步思考（Let's think step by step）"
- **Self-Consistency**：多次采样取多数投票，提高推理准确率
- **Tree-of-Thought（ToT）**：将思维链扩展为思维树，探索多条推理路径

例如：
```
问：一个商店有35个苹果。如果用掉20个做果汁，又进货25个，现在有多少个苹果？

让我们一步一步思考：
1. 开始有35个苹果
2. 用掉20个：35 - 20 = 15个
3. 进货25个：15 + 25 = 40个
答案是40个苹果。
```

### 6.4.4 System Prompt 的作用

**System Prompt（系统提示）** 是在对话开始前设置的隐藏指令，用于定义模型的行为准则：

- 设定角色和人格（"你是一个专业的法律顾问"）
- 定义回答风格（简洁/详细、正式/口语化）
- 设置安全边界（拒绝回答的内容类型）
- 提供领域知识和背景信息

System Prompt 对于构建垂直领域的 AI 应用至关重要。

### 6.4.5 最佳实践与技巧

1. **明确具体**：指令越清晰，输出越准确
2. **结构化输入**：使用分隔符、标签组织 Prompt
3. **指定输出格式**：用 JSON Schema 或示例明确格式要求
4. **迭代优化**：Prompt 需要反复测试和调整
5. **控制温度参数**：temperature 低 → 更确定性，temperature 高 → 更有创意
6. **避免否定指令**：说"请只讨论技术话题"而非"不要讨论非技术话题"

## 6.5 检索增强生成（RAG）

### 6.5.1 为什么需要 RAG

LLM 存在两个根本性问题：

- **知识截止（Knowledge Cutoff）**：模型只知道训练数据截止日期之前的信息
- **幻觉（Hallucination）**：模型会自信地生成看似合理但实际上错误的内容

**RAG（Retrieval-Augmented Generation，检索增强生成）** 通过引入外部知识源来解决这些问题，让模型"有据可依"。

### 6.5.2 RAG 的基本流程

```
用户提问 → 检索（Retrieve）→ 增强（Augment）→ 生成（Generate）
```

1. **检索阶段**：将用户问题转换为向量，从知识库中检索最相关的文档片段
2. **增强阶段**：将检索到的文档片段拼接到 Prompt 中作为上下文
3. **生成阶段**：LLM 基于增强后的 Prompt 生成回答

### 6.5.3 向量数据库

向量数据库是 RAG 系统的核心基础设施，用于存储和高效检索文档的向量表示：

| 数据库 | 特点 |
|--------|------|
| **Chroma** | 轻量级，适合开发和小规模项目 |
| **Pinecone** | 云端托管，开箱即用 |
| **Milvus** | 开源，支持十亿级别向量 |
| **Weaviate** | 支持混合搜索（向量 + 关键词） |
| **FAISS** | Meta 开源的高效向量检索库 |
| **Qdrant** | Rust 编写，高性能 |

### 6.5.4 Embedding 模型

Embedding 模型负责将文本转换为高维向量表示，是 RAG 的"眼睛"：

- **OpenAI text-embedding-3**：商业 API，效果好
- **BGE 系列**（智源研究院）：中文效果优秀的开源模型
- **GTE 系列**（阿里巴巴）：通用文本表示模型
- **Jina Embeddings**：支持超长文本（8K tokens）
- **Cohere Embed**：多语言支持好

### 6.5.5 RAG 的优化方向

基础 RAG 存在检索质量不高、上下文利用不充分等问题，优化方向包括：

- **查询改写（Query Rewriting）**：用 LLM 改写用户问题以提高检索效果
- **混合检索（Hybrid Search）**：结合向量检索和传统关键词检索（BM25）
- **重排序（Re-ranking）**：用交叉编码器对检索结果重新排序
- **文档分块策略**：优化文档的切分方式（按段落、按语义、递归分块）
- **多跳检索（Multi-hop Retrieval）**：对于复杂问题进行多轮检索
- **Graph RAG**：结合知识图谱进行结构化检索（微软 2024 年提出的方法尤其受关注）
- **Agentic RAG**：让 Agent 自主决定何时检索、检索什么
- **自适应 RAG（Adaptive RAG）**：动态判断是否需要检索，减少不必要的检索开销
- **Self-RAG**：模型自主评估检索内容的相关性和准确性，进行自我验证

> **注**：随着超长上下文模型（如 Gemini 1.5 Pro 的 100 万 tokens）的出现，部分简单场景可能不再需要 RAG。但对于需要实时更新的知识、海量文档检索等场景，RAG 仍然是不可替代的方案。

## 6.6 AI Agent

### 6.6.1 什么是 AI Agent

**AI Agent（智能体）** 是以 LLM 为"大脑"，具备**自主规划、工具调用、记忆和反思**能力的 AI 系统。与简单的聊天机器人不同，Agent 能够：

- **规划（Planning）**：将复杂任务分解为子任务
- **工具调用（Tool Use）**：调用外部 API、搜索引擎、代码执行器等
- **记忆（Memory）**：维护短期和长期记忆
- **反思（Reflection）**：评估自己的行为并进行调整

Agent 的核心公式：**Agent = LLM + 规划 + 工具 + 记忆**

### 6.6.2 ReAct 框架

**ReAct（Reasoning + Acting）** 是最经典的 Agent 框架，交替进行推理和行动：

```
思考（Thought）→ 行动（Action）→ 观察（Observation）→ 思考 → 行动 → ...
```

示例流程：
```
用户：北京今天的天气怎么样？

思考：我需要查询北京的实时天气信息。
行动：调用天气API，参数：城市=北京
观察：API返回 - 晴，气温25°C，湿度40%
思考：我已经获得了天气信息，可以回答用户了。
回答：北京今天天气晴朗，气温25°C，湿度40%，适合外出活动。
```

### 6.6.3 Function Calling / Tool Use

**函数调用（Function Calling）** 是让 LLM 能够调用外部工具的关键机制：

- 模型根据用户意图，**决定调用哪个函数**以及**传递什么参数**
- 函数执行结果返回给模型，模型再基于结果生成最终回答
- OpenAI、Claude、Gemini 等主流模型都原生支持 Function Calling
- **MCP（Model Context Protocol）**：Anthropic 于 2024 年发布的开放标准协议，为 AI 模型与外部工具、数据源之间提供了统一的连接方式。相比传统 Function Calling，MCP 提供了更灵活的资源管理和双向通信机制，已被 Claude、VSCode、Cursor 等广泛采用，正在成为行业标准

典型工具类型：搜索引擎、代码解释器、数据库查询、文件操作、API 调用等。

### 6.6.4 多智能体系统（Multi-Agent）

当单个 Agent 难以处理复杂任务时，可以让**多个 Agent 协作**：

- **角色分工**：不同 Agent 扮演不同角色（程序员、测试员、产品经理）
- **讨论与辩论**：多个 Agent 互相讨论，得出更优解
- **层级结构**：管理者 Agent 分配任务给执行者 Agent
- **典型框架**：CrewAI、AutoGen、MetaGPT、ChatDev、LangGraph（LangChain 的 Agent 框架）、OpenAI Agents SDK

### 6.6.5 典型 Agent 应用

| 应用 | 说明 |
|------|------|
| **Devin** | Cognition 推出的 AI 软件工程师，能自主完成开发任务 |
| **Cursor / Windsurf** | AI 驱动的代码编辑器，深度集成 Agent 能力 |

| **LangChain Agent** | 基于 LangChain 框架构建的可定制 Agent |
| **AutoGPT** | 早期的自主 Agent 尝试，探索了自动化任务执行 |
| **OpenAI Assistants** | OpenAI 的 Agent API，支持代码解释器、文件检索等 |

## 6.7 模型部署与优化

### 6.7.1 模型量化（Quantization）

量化是将模型权重从高精度（如 FP16）转换为低精度（如 INT8、INT4）的技术，大幅减少显存占用和推理成本：

| 方法 | 精度 | 说明 |
|------|------|------|
| **FP16/BF16** | 16位 | 标准训练精度，性能损失极小 |
| **INT8** | 8位 | 显存减半，适合大多数场景 |
| **INT4** | 4位 | 显存降至1/4，适合消费级 GPU |
| **GPTQ** | 4位 | 基于二阶信息的后训练量化，精度较高 |
| **AWQ** | 4位 | 激活感知量化，保护重要权重通道 |
| **GGUF** | 混合 | llama.cpp 使用的格式，支持 CPU 推理 |

量化使得在消费级显卡（如 RTX 4090）甚至 CPU 上运行大模型成为可能。

### 6.7.2 知识蒸馏（Knowledge Distillation）

知识蒸馏是让小模型（学生）学习大模型（教师）的行为：

- **输出蒸馏**：学生模型学习教师模型的输出概率分布
- **特征蒸馏**：学习教师模型中间层的特征表示
- **应用场景**：用 GPT-4 的输出训练小模型（如 Alpaca 用 GPT-3.5 输出训练 LLaMA 7B）
- **优势**：可以大幅降低部署成本，同时保留大模型的大部分能力

### 6.7.3 LoRA / QLoRA 高效微调

**LoRA（Low-Rank Adaptation）** 是当前最流行的高效微调方法：

- **核心思想**：冻结原始模型权重，只训练低秩分解的小矩阵（Adapter）
- **参数效率**：通常只需训练原始参数量的 0.1%~1%
- **QLoRA**：在 LoRA 基础上结合 4-bit 量化，单张消费级 GPU 即可微调大模型
- **实际效果**：在特定任务上可以达到全量微调 90% 以上的效果

```
# 使用 LoRA 微调的示例概念
原始权重 W (冻结) + 低秩矩阵 ΔW = A × B
其中 A 的形状为 (d, r)，B 的形状为 (r, d)，r << d
训练参数量：2 × d × r（远小于 d × d）
```

### 6.7.4 推理框架

| 框架 | 特点 | 适用场景 |
|------|------|---------|
| **vLLM** | PagedAttention、连续批处理，吞吐量极高 | 生产环境部署 |
| **Ollama** | 一键安装运行，用户体验好 | 本地开发和体验 |
| **llama.cpp** | 纯 C++ 实现，支持 CPU 推理 | 边缘设备、无 GPU 环境 |
| **TGI** | HuggingFace 出品，支持多模型并发 | 云端服务 |
| **SGLang** | 结构化生成优化，速度快 | 需要结构化输出的场景 |
| **TensorRT-LLM** | NVIDIA 优化，充分利用 GPU 特性 | NVIDIA GPU 环境 |
